Session Web Handshake Standard

Securing communications between client and servers using short lived hybrid cryptography

Introduction

Security has always been a problem for web development. We use "HTTP" as the default protocol which is not secured as whatever we sent thru it will be sent exactly... just like your password in plain text! with that in mind, imagine someone looking into your network trafic and stealing your credentials so easily. it can be your co-worker, a hacker, a staff the company whose service you are accessing and so on...
Here comes "HTTPS" and "VPN" to the rescue but what are they and how do they work? what are there advantages and limitations? I will not pretend to be an expert on this and explain these 2 technologies as I understand it in semi-layman's term

HTTPS
  • The 'S' in "HTTPS" basically means secured thus also called HTTP-Secured.
  • it works is by adding a layer of encryption from the browser to the target service owner by using SSL certificates which includes a Public that enables clients to send data to your APIs encrypted while you hold the private key to decrypt them.
  • The certificate is only used for the initial handshake in which afterwards, a new set of keys will be genrated called "session keys" which will allow our APIs to also send back to them encrypted.
  • This certificate is also held (and mostly produced) by a Certificate Authority (CA) that investigates your company and site to ensure the domain's authenticity.
  • The browsers we use verify with the CA's to detect whether the site the are in is secured or not. If verified, the browser makes it visble to the user by showing a padlock symbol and green highlighted URL to indicate an all-clear sign, red if unverified.
VPN
  • Stands for "Virtual Private Network" and as the name suggests its not really private but is virtualized because you do not use a private line but a public line wherein the communications is secured in some way.
  • VPNs simulates a network infrastructure which could help mask the origin of the clients and use a local IP of the server instead for making the calls (Tunneling). This is what makes it popular such that it allows people to access services not available in their region
  • VPNs secure the transmission by encrypting the data from your device to their server making the call on your behalf

Everything seems secured using just these! So why build a new way to secure the transmission data?
  • Remember when I mentioned the parties involved? The CA (Which may have ties to governments), the 3rd party VPNs (who can still look at your data before they encrypt it) and the Browser (i.e. Chrome "by Google" which collects our data for its business)?
  • We as humans rely on a higher authority and believe immediately that such entities will have our best interests at heart. In reality, the only security is a security which you control. Adding more entities to manage your security for you only adds more vectors for attack.
  • Cost! If we do not need HTTPS and VPNs but still want to secure our data, what do we use? We need something that can work with HTTP alone or combined with these secured process as an additional measure
  • VPN is not end to end encryption. They may start encrypting from your request up to the point their server makes the requestr, but that request is no longer encrypted unless it by them otherwise how will they make the proper call?
  • HTTPS encryption is managed in the SESSION Layer of the OSI model. This is where the browser starts to encrypt your requests. But what is stopping a compromised or malicious browser from sending these data to themselves first? same as well for your 3rd party webserver app?

So what is SWHS

The Session Web Handshake Standard is not a new technology nor is it a concept that is unique and novel. SWHS is a proposed standard for developing services for data sensitive clients which embraces the current best practices utilized by existing web security protocols. It tries to address the following issues that is common in most developers who will work in security
  • What if we will only use HTTP/TCP? Is there a way we can secure data coming in and out from our clients to us?
  • We already use HTTPS or VPN or Both! but I do not trust these entities and would like to add another layer of security to my clients.
  • I want the handshake to be unpredictable, such that the handshake keys change for each new connection
  • I want the encryption to take place in the Application layer for both our managed server an client apps
The best solution is always the one where only the 2 parties have full control of the data. Encrpytion should happen in the Applications level before we send the data to the network(client) and after we have fully received the data from the network (server)
